Auth Login Flow Analysis
========================

Flow
----
1. Generate a PKCE pair (code_verifier + code_challenge) and a random state token
2. Start a temporary HTTP server on 127.0.0.1 with a random port
3. Build an Auth0 /authorize URL with the PKCE challenge, redirect to http://localhost:{port}/callback
4. Print the URL to stderr, then call webbrowser.open()
5. Poll in a loop waiting for the callback, with a 120-second timeout
6. Callback handler validates state, captures the authorization code
7. Exchange the code + code_verifier for tokens via POST to Auth0 /oauth/token
8. Extract email from the JWT id_token payload
9. Save tokens via token_store (keyring or file fallback)

Assumptions
-----------
- Browser on same machine: The redirect URI is http://localhost:{random_port}/callback —
  the browser must be able to reach the localhost callback server
- GUI environment: webbrowser.open() expects a display (though the URL is also printed as
  fallback text)
- Port availability: Binds to 127.0.0.1:0 (random free port), so the redirect URI is
  different every time — Auth0 must allow wildcard or dynamic localhost redirect URIs
- Single-use: The callback server shuts down after one response — no retry within the
  same login attempt
- Network access: Needs outbound HTTPS to Auth0 for the token exchange
- 120-second window: User must complete browser login within 2 minutes

Weaknesses
----------
- No headless/SSH support — fails silently if no browser is available
- The random port means Auth0 must be configured to accept http://localhost:*/callback or
  a broad set of localhost redirects (Auth0 supports http://localhost with any port for
  native apps by default, so this works)
- No way to resume or retry if the callback is missed — must restart the entire flow
